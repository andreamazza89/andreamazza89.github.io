Finally, I started looking into TestDrivenDevelopment. In a nutshell, TDD is a (relatively) new way to approach writing code, which is part of Agile methodologies. Rather than writing a piece of code and then devising a test to check if it works, the opposite is done. That is, first a test is written, then the code to make it pass. The test defines an expectation on the code, for example one could say "the function shout is expected to return the string 'ARGH!' when called". The code is then written to satisfy the test.
This has a number of advantages; using small and clear tests allows us to keep track of what is required from the program and avoid going off the rails with additional features that were not on the brief. It also helps dealing with few small problems at one time, as we work on one test at a time. A great side effect of this methodology is the fact that the suite of tests also double up as documentation on the code.
I have not officially started adopting this yet, but I shall soon 'force' myself to only use TDD. It will feel backwards and slower at first, but I will stick with it and write some clean, well documented code.
