The focus of this week has been command line and Git. I am only going to talk about the former.

In general, command line describes one way in which the user interacts with the machine. When using command line, there are no graphics, only lines of text, one after the other. Each line is either the computer 'talking' to the user, or vice-versa. For instance, the computer might say something like: "I am currently looking at the Downloads folder and am ready for commands." And the user might say: "Add a folder called Pictures and move all my pictures into it". Cool no? Another way the user can interact with the computer is using a graphical interface, which is the most common way (the one we all know), as it is easier to pick up and looks nicer (not as cool though!).

Now for a bit more jargon. Both the command line and graphical interfaces can be referred to as "shell". The name is not accidental: the core level of computer operations is called "Kernel" and is very difficult for a human to 'talk' in Kernel language. So the shell is wrapped around the Kernel and allows us, the human, to talk to the Kernel and get it to do things for us. The shell is really just a translator. Staying on the methaphor, there are different languages we may speak (i.e. command-line or graphical user interface), and different translators to translate each language into Kernel language. The most used command-line dialect (sorry, I really like methaphors, and there is more than one command-line language!) is called Bash, which is what we have been learning this week. I made a little drawing to exemplify the human-shell-kernel concept.

The command line used to be the only way we could interact with computers, however, fast forward a few decades and there are some pretty attractive alternatives, namely the various graphical user interfaces we are used to, be it the latest Window or OSX.

The question is: why do we use command line? Is it better? Is it worse? I think the answer is - it depends.

For the average user, the graphical interface is incredibly easy to pick up , intuitive and requires very little knowledge of its 'language': we know we can move the pointer to a file, double click on it to open it, or drag and drop it on a different folder, or right click for more options, etc.
On the other side, we have the command line, which requires the user to learn the various arcane commands like 'cp' or 'rm' or 'cd' to achieve the same results.

So far not so good for command-line. Why go through the pain of learning all the syntax? A world of programmer-perks awaits. Here are a few:

  • Faster processing speed. This might seem useless if you think about copy/pasting one file, but when dealing with vast amounts of files to process, a shell that has no graphics to worry about is a faster shell.
  • No mouse, faster human. Although it takes learning the syntax and being a quick typer, having hands stuck to the keyboard and not having to touch the mouse speeds thing up.
  • Fewer steps to achieve the same thing. This can be explained with an example: suppose we want to move a file "TEST.TXT" into its parent folder.Here are the steps required in command line:
    a) type mv TEST.TXT ..
    And these using a graphical user interface (one of the ways it could be done):
    a) right-click on the file, press 'cut'
    b) navigate to the parent folder
    c) right-click somewhere in the folder, find a bit of empty space, right-click and press 'paste'
  • A command-line environment like Bash also allows you to use scripts to automate processes.

I am sure there are numerous other advantages as well as disadvantages, but I am going to stop here. The main point is that all these advantages are good for programmers but useless for non-programmers (or at least the gains are not worth the effort), which is why programmers still use command-line and others do not!
